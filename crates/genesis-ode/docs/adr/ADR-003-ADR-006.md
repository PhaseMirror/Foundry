# Lets provide the coding agent with phased ADR production grade implementation plans including detailed instructions, file scaffold and test harness.

Yes вЂ” the right next artifact is a **phased, production-grade implementation plan** for a coding agent, with explicit ADR workflow, repository scaffold, interfaces, invariants, and test harness requirements. Good ADR practice is to keep each record focused on one significant decision, store them in version control, and include consistent sections such as context, decision, status, and consequences; architecture tests can then enforce those decisions in code.[^1][^2][^3]

## Implementation plan

Below is a coding-agent-ready plan for implementing the Exploder/Builder stack as governed Lane A infrastructure, while keeping the scalar core untouched and the ADR trail append-only. The plan assumes ADR-001/002 are already authoritative and ADR-003 is the admission/control record for Exploder/Builder and the П„ tether.[^4][^5]

### Phase 0 вЂ” ADR baseline

**Goal:** establish decision records first, before code. ADRs should live in-repo, use a consistent template, and remain immutable except for status/supersession updates; if the approach changes, add a new ADR rather than rewriting history.[^6][^4]

**Coding agent tasks**

- Create `docs/adr/`.
- Add:
    - `ADR-001.md` and `ADR-002.md` as imported references if not already present.
    - `ADR-003-exploder-builder-twin.md` as the controlling Lane A admission record.
    - `ADR-004-perturbation-families.md` for metallurgical perturbation policy.
    - `ADR-005-shrapnel-schema-and-certificates.md` for canonical JSON contracts.
    - `ADR-006-test-harness-and-quality-gates.md` for required tests and release gates.
- Use one decision per ADR, with:
    - Title
    - Status
    - Context
    - Decision
    - Consequences
    - Related ADRs
    - Confidence / review status
This follows widely used ADR guidance and keeps later code traceable to named decisions.[^7][^3][^8]

### Phase 2 вЂ” Canonical data contracts

**Goal:** implement schemas first, because every later component reads or writes these artifacts. The coding agent should treat the shrapnel map and resistance certificate as stable interfaces, not incidental dictionaries.

**Required models**

- `ShrapnelFragment`
- `ShrapnelMap`
- `ResistanceCertificate`
- `ExploderRunManifest`
- `BuilderProposal`
- `GovernanceDecision`

**Minimum fields**

```python
# shrapnel.py
target_id: str
baseline_intent: str
test_suite: list[str]
observed_drift: dict[str, float | int]
fragility_class: str
shrapnel_fragments: list[ShrapnelFragment]
resistance_certificate: ResistanceCertificate | None
tether_tension: float
tier: Literal["S", "I"]
provenance: dict[str, Any]
```

**Rules**

- All outputs must be serializable to canonical JSON.
- Every object must carry:
    - `schema_version`
    - `created_at`
    - `run_id`
    - `source_adr_ids`
- `prime_signature` is metadata only, never a driver of scalar dynamics.
- `tier` is mandatory and explicit.
- Unknown fields should fail validation in tests.


### Phase 3 вЂ” Exploder engine

**Goal:** create a deterministic, replayable exploration harness for metallurgical fatigue first. The coding agent must keep Exploder side-effect free: simulation only, no write path into production state.

**Implement**

- `PerturbationFamily` interface:
    - `name`
    - `parameter_ranges`
    - `apply(surface_state) -> surface_state`
    - `expected_bounds`
- Initial perturbation families:
    - `amplitude_ramp`
    - `frequency_shift`
    - `phase_shift`
    - `timing_jitter`
    - `drag_spike`
    - `dual_symbolic_numeric`
- `ExploderEngine.run(target, suite, policy)`:
    - loads baseline state
    - executes perturbation schedule
    - measures drift against $\varepsilon_X$
    - classifies fragility
    - computes coverage
    - emits shrapnel map JSON and optional resistance certificate

**Classification labels**
Start with 6 canonical labels:

- `impedance-spike-precursor`
- `recovery-deficit-amplification`
- `fragile-under-coupling`
- `time-delayed-failure`
- `dual-disagreement`
- `robust-under-adversarial-stress`


### Phase 4 вЂ” Tether policy

**Goal:** encode П„ as policy, not prose. The coding agent should implement П„ as a pure function plus policy thresholds in config.

**Metric**

$$
\tau = \left(\frac{\text{coverage}}{\text{required\_coverage}}\right)\times\left(1-\frac{\lVert\Delta_{\text{drift}}\rVert}{\varepsilon_X}\right)
$$

**Implement**

- `tether.metric.compute_tau(coverage, required_coverage, drift_norm, epsilon_x) -> float`
- `tether.policy.can_builder_advance(shrapnel_map, policy) -> bool`
- `tether.novelty.detect_novelty(run_manifest, prior_runs) -> NoveltyAssessment`

**Policy defaults**

- `tau_threshold = 0.70`
- `coverage_min = 0.80`
- `novelty_freeze = True`
- freeze Builder on:
    - unseen fragility class
    - unseen perturbation family interaction
    - drift above `epsilon_x`
    - unresolved dual disagreement


### Phase 5 вЂ” Builder engine

**Goal:** ensure Builder can only assemble from certified fragments and must preserve provenance.

**Implement**

- `BuilderEngine.propose(fragments, intended_use, policy) -> BuilderProposal`
- admission checks:
    - every fragment has a shrapnel map entry
    - every fragment is not banned/quarantined for this use
    - certificate scope covers intended use
    - П„ above threshold
    - all inherited gaps are attached to proposal
- Builder never upgrades status by itself; it only emits a proposal requiring governance review.


### Phase 6 вЂ” Governance adapters

**Goal:** map artifacts to explicit review and rollback actions.

**Implement**

- `governance.rollback.evaluate(shrapnel_map) -> list[RollbackTrigger]`
- `governance.review.summarize(builder_proposal) -> GovernancePacket`
- `governance.tiering.enforce(packet) -> TieredDecision`

**Rules**

- `Unstable Shrapnel` must map directly to rollback review conditions.
- Any attempt to use Exploder output as core evidence should fail a policy check.
- Governance packets must include:
    - shrapnel map
    - certificate
    - П„ value
    - novelty assessment
    - provenance summary


## Test harness

### Unit tests

The coding agent should implement these first:

- Schema validation rejects missing `tier`, `run_id`, or `schema_version`.
- `compute_tau()` handles:
    - nominal values
    - coverage > required coverage
    - drift = 0
    - drift >= epsilon_x
    - negative input rejection
- Perturbation families stay within declared parameter ranges.
- Provenance merger preserves inherited gaps and source fragment IDs.
- Fragility classifier is deterministic for fixed seeds.


### Integration tests

Then implement end-to-end tests:

1. **Metallurgical Exploder run**

- Baseline surface state in fixture.
- Run `amplitude_ramp + phase_shift + dual_symbolic_numeric`.
- Assert:
    - shrapnel JSON produced
    - fragility class assigned
    - drift recorded
    - П„ computed
    - certificate present or explicitly absent with reason

2. **Builder admission**

- Feed certified fragments with П„ above threshold.
- Assert proposal succeeds.
- Feed one fragment lacking certificate coverage.
- Assert proposal rejected with explicit reason.

3. **Rollback mapping**

- Feed shrapnel map labeled `impedance-spike-precursor`.
- Assert governance produces rollback review trigger.

4. **Novelty freeze**

- Feed run with unseen fragility class.
- Assert Builder frozen regardless of П„.


### Architecture tests

Use architecture tests to enforce the constitutional boundaries:

- `builder` cannot import `core` mutation utilities.
- `exploder` cannot call deployment or production adapters.
- `governance` can read artifacts but not mutate scalar core state.
- `schemas` may be imported everywhere; no reverse dependency from `schemas` into engines.
This is exactly the kind of import-boundary rule that architecture-test frameworks are designed to enforce.[^1]
