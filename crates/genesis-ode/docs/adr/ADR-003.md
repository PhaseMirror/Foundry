## Context
To evolve the Genesis ODE ecosystem without compromising the core, we need a twin-system approach: the **Exploder** (adversarial stress-testing) and the **Builder** (constructive assembly). These must operate as governed Lane A infrastructure.

## Decision
We implement the Exploder/Builder stack with the following constraints:

1. **Separation of Concerns:** 
   - **Exploder:** Deterministic, side-effect-free simulation engine. It applies perturbations to a copy of the surface state and measures the response.
   - **Builder:** Provenance-preserving assembly engine. It proposes state updates based on certified fragments from the Exploder.
2. **Immutable Core:** Neither system may mutate the `genesis_core` directly.
3. **Artifact-Driven Governance:** Communication between Exploder, Builder, and Governance layers happens via canonical JSON artifacts (`ShrapnelMap`, `ResistanceCertificate`).
4. **The τ Tether:** A quantitative metric (τ) gates the Builder's ability to propose changes.

## Consequences
- **Positive:** Enables safe experimentation. Automates quality gates.
- **Negative:** Increased complexity in data contract management.
- **Tradeoff:** Safety and auditability over rapid, ungoverned iteration.
