# ADR-007: Multiplicity Metadata & Prime-Based Encoding

**Status:** PROPOSED  
**Date:** 2026-05-20  
**Author:** Gemini CLI Agent  
**Dependency:** ADR-001, ADR-003, ADR-005  

## Context
The Genesis Governance ecosystem requires a method to encode structural invariants and cross-substrate relationships using prime signatures (Multiplicity) without compromising the integrity of the scalar core dynamics $C_X(t)$.

## Decision
We implement a **Multiplicity Metadata Overlay** with the following constraints:

1.  **Metadata Only:** Prime signatures, exponents, and multiplicity vectors are strictly metadata. They may be carried by `SurfaceState` or `ShrapnelFragment` but MUST NOT influence the numerical evolution of the ODE core.
2.  **Sparse Encoding:** The encoding should prioritize sparsity (small changes in state → sparse updates in signature).
3.  **Module Quarantine:** All encoding/decoding logic must reside in a dedicated `multiplicity/` module, which is subject to the same isolation rules as the Exploder/Builder (read-only access to core).
4.  **Schema Enforcement:** Multiplicity metadata must follow the `MultiplicityEncoding` Pydantic model defined in `types.py`.

### Multiplicity Schema:
- `prime_signature`: List of active prime factors.
- `exponent_vector`: Dictionary mapping primes to their multiplicity (exponents).
- `sparsity_index`: A metric of encoding density.
- `locality_score`: Measures how well the encoding preserves state proximity.

## Consequences
- **Positive:** Enables higher-order analysis of simulation results. Formalizes the innovation budget for multiplicity research.
- **Negative:** Adds complexity to the `SurfaceState` model and artifact serialization.
- **Tradeoff:** Analytical depth over model simplicity.
