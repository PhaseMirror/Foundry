# PWEH External Interface: Checkpoint Verification Guide

**L0 Boundary Anchor:** The PWEH receipt schema and Lean root (`substrates/proofs/PwehReceipt.lean`) are governed artifacts under the L0 boundary.

**Target Audience:** External Verifiers (e.g., DecisionAssure)
**Purpose:** Confirming the operational continuity and integrity of the system without requiring knowledge of internal logic or mathematical formulas.

## 1. What is a Checkpoint?
A checkpoint is a secure receipt generated by the system at key moments. It serves as cryptographic proof that all actions taken up to that point were authorized, mathematically bounded, and correctly recorded.

## 2. How the Checkpoint is Structured
When you receive a checkpoint receipt, it contains five key pieces of information in a standard JSON format:
1. **Integrity State:** A unique digital fingerprint representing the entire history of the system's safe operation up to this moment.
2. **Last Action:** A plain-text description of the exact move or command that was just executed.
3. **Policy Root:** A digital fingerprint of the specific ruleset (policy) that was active and governing the system during this action.
4. **Safety Certificate:** A digitally signed confirmation from the system's core monitor proving that the action did not violate any structural safety limits.
5. **Resonance Score:** A health metric indicating the system's stability. A value near 1.0 indicates perfect structural health.

## 3. How to Verify Continuity
To confirm the system has operated continuously and safely between any two checkpoints (Checkpoint A and Checkpoint B), you must:

1. **Validate the Formats:** Feed the receipt into the provided reference parser to ensure all digital fingerprints are properly formatted (64-character hexadecimal codes) and all required fields are present.
2. **Check the Safety Certificate:** Confirm that the safety certificate is present and valid, indicating that the system's core monitor approved the action.
3. **Chain the Fingerprints:** Ensure that the Integrity State fingerprint of Checkpoint B correctly accounts for Checkpoint A's state plus the "Last Action" taken. (The provided parsing and verification tools handle this cryptographic chaining automatically).
4. **Monitor the Health:** Verify that the Resonance Score remains near 1.0, indicating no slow degradation or drift in the system's strict safety boundaries.

If the parser approves the receipt and the fingerprints link together without breaking, you have mathematical certainty that the system's timeline is continuous, tamper-proof, and fully compliant with its governed rules.

## 4. Exact JSON Schema
The exact structure transmitted to verifiers must adhere to the following schema:
```json
{
  "s_integrity": "<64-char lowercase hex>",
  "last_prime_move": <positive integer (prime)>,
  "policy_root_hash": "<64-char lowercase hex>",
  "crmf_certificate": "<64-char lowercase hex>",
  "lambda_m_resonance_score": <number 0.0-1.0>
}
```

## 5. Examples

### Minimal Valid Example
```json
{
  "s_integrity": "1234567890abcdef1234567890abcdef1234567890abcdef1234567890abcdef",
  "last_prime_move": 43,
  "policy_root_hash": "abcdef1234567890abcdef1234567890abcdef1234567890abcdef1234567890",
  "crmf_certificate": "1111111111111111111111111111111111111111111111111111111111111111",
  "lambda_m_resonance_score": 0.992
}
```

### Invalid Examples (Failure Cases)

**Case 1: Invalid Hash Length (Too Short)**
```json
{
  "s_integrity": "short",
  "last_prime_move": 3,
  "policy_root_hash": "bbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbb",
  "crmf_certificate": "cccccccccccccccccccccccccccccccccccccccccccccccccccccccccccccccc",
  "lambda_m_resonance_score": 0.5
}
```
*(Fails because `s_integrity` is not exactly 64 characters long).*

**Case 2: Resonance Score Out of Bounds**
```json
{
  "s_integrity": "aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa",
  "last_prime_move": 3,
  "policy_root_hash": "bbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbb",
  "crmf_certificate": "cccccccccccccccccccccccccccccccccccccccccccccccccccccccccccccccc",
  "lambda_m_resonance_score": 1.5
}
```
*(Fails because `lambda_m_resonance_score` exceeds the maximum boundary of 1.0).*

**Case 3: Invalid Hash Character Format (Uppercase or '0x' prefix)**
```json
{
  "s_integrity": "0x1234567890abcdef1234567890abcdef1234567890abcdef1234567890ab",
  "last_prime_move": 43,
  "policy_root_hash": "ABCDEF1234567890ABCDEF1234567890ABCDEF1234567890ABCDEF1234567890",
  "crmf_certificate": "cccccccccccccccccccccccccccccccccccccccccccccccccccccccccccccccc",
  "lambda_m_resonance_score": 0.5
}
```
*(Fails because hashes must be strictly lowercase hex without '0x' prefixes).*
