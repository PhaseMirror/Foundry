## ✅ RESOLVED: diff (jsdiff) Denial of Service Vulnerability

**Status:** ✅ PATCHED  
**Severity:** MEDIUM - Denial of Service vulnerability  
**Date Identified:** January 22, 2026  
**Date Resolved:** January 22, 2026  
**Affected Package:** diff >= 6.0.0, < 8.0.3  
**Patched Version:** 8.0.3

### Vulnerability Description

The `parsePatch` and `applyPatch` methods in jsdiff can enter an infinite loop when parsing patches with filename headers containing line break characters (`\r`, `\u2028`, or `\u2029`), leading to memory exhaustion and process crash.

### Attack Vector

1. Attacker provides a malicious patch with special line break characters in filename headers
2. Application calls `parsePatch()` or `applyPatch()` with the malicious patch
3. Infinite loop triggered, consuming memory without limit
4. Process crashes due to out-of-memory condition

**Note:** A large payload is NOT needed - size limits do not provide protection.

### Affected Dependencies

This vulnerability was introduced transitively through:
- `mocha@11.7.5` → `diff@^7.0.0`
- `@nomicfoundation/hardhat-toolbox-mocha-ethers@3.0.2` → ... → `diff@^7.0.0`

### Resolution

✅ **Override Applied:** Added `"diff": ">=8.0.3"` to `pnpm.overrides` in package.json  
✅ **Verification:** Confirmed diff 8.0.3 installed across all dependencies  
✅ **Impact:** Development/testing environment only (Hardhat, mocha)

**Fixed in:** [PR/Commit reference here]  
**References:** 
- [CVE Advisory](https://github.com/advisories/GHSA-h6ch-v84p-w6p9)
- [Fix PR kpdecker/jsdiff#649](https://github.com/kpdecker/jsdiff/pull/649)

## ⚠️ ACTIVE: Elliptic Package Cryptographic Vulnerability

**Status:** ⚠️ UNPATCHED - No official fix available  
**Severity:** HIGH - Cryptographic vulnerability leading to potential private key exposure  
**Date Identified:** January 22, 2026  
**CVE:** Pending  
**Affected Package:** elliptic <= 6.6.1 (all known versions)

### Vulnerability Description

The ECDSA implementation in the Elliptic package generates **incorrect signatures** when an interim value of 'k' (computed per RFC 6979 step 3.2) contains leading zeros. This occurs because the byte-length of 'k' is incorrectly computed, causing truncation during the signature generation process.

### Attack Vector

An attacker who obtains:
1. A faulty signature generated by a vulnerable version of Elliptic
2. A correct signature for the same inputs

Could potentially derive the **secret private key** through cryptanalysis.

### Impact Assessment

#### Direct Impact
- **Broken Transactions:** Legitimate ECDSA signatures may be invalid
- **Key Exposure Risk:** Private keys could be compromised under specific conditions
- **Communication Integrity:** Cryptographic communications may be compromised

#### Affected Dependencies in Lambda-Proof

This vulnerability is introduced transitively through:

1. `@nomicfoundation/hardhat-ignition-ethers@0.15.17` → ... → `elliptic@6.6.1`
2. `@nomicfoundation/hardhat-network-helpers@1.1.2` → ... → `elliptic@6.6.1`
3. `@nomicfoundation/hardhat-toolbox-mocha-ethers@3.0.2` → ... → `elliptic@6.6.1`
4. `@nomicfoundation/hardhat-verify@3.0.8` → ... → `elliptic@6.6.1`
5. `circomlibjs@0.1.7` → ... → `elliptic@6.6.1`
6. `hardhat-gas-reporter@2.3.0` → ... → `elliptic@6.6.1`
7. `solidity-coverage@0.8.17` → ... → `elliptic@6.6.1`

### Current Status

- **Official Patch:** ❌ None available (elliptic@6.6.1 is the latest version)
- **Community Fix PR:** 🔄 [PR #345](https://github.com/indutny/elliptic/pull/345) - "fix: preserve leading zeros in k-value during ECDSA signing" (Open, not merged)
- **Maintained Forks:** ❌ None identified
- **Last elliptic Update:** November 13, 2024

### Mitigation Strategy

#### Immediate Actions Taken

1. ✅ **Override Constraint Added:** `pnpm.overrides` includes `"elliptic": ">=6.6.1"` to prevent downgrades
2. ✅ **Documentation:** This advisory file created for tracking
3. ⚠️ **Monitoring:** Track upstream PR #345 for merge status
4. ✅ **Monitoring Script:** Created `scripts/check-elliptic-security.mjs` for automated checks

#### Recommended Mitigations

##### For Development Environment (Current Usage)
Since these packages are primarily used in **development/testing environments** (Hardhat tooling), the risk is mitigated:

- ✅ **Hardhat Development:** Local testing and development use only
- ✅ **No Production Key Exposure:** Development private keys are not used for production funds
- ✅ **Gas Reporting/Coverage:** Non-critical tooling operations

##### For Production Considerations

⚠️ **CRITICAL:** Do NOT use elliptic-dependent packages for:
- Generating production private keys
- Signing production transactions
- Any cryptographic operations involving real assets

Instead, use:
- `@noble/curves` (modern, audited alternative)
- `@noble/secp256k1` (for Bitcoin/Ethereum curves)
- Hardware wallets for key management
- ethers.js/viem native signing (which use different crypto implementations)

### Action Items

- [ ] Monitor [elliptic PR #345](https://github.com/indutny/elliptic/pull/345) for merge status
- [ ] Check for elliptic releases > 6.6.1 monthly (via `pnpm check:security:elliptic`)
- [ ] Consider migrating dev tooling to packages that use @noble/curves when available
- [ ] Update this advisory when patch becomes available
- [ ] Review circomlibjs alternatives if available

### Workaround for circomlibjs

`circomlibjs@0.1.7` is a production dependency. Since it uses elliptic for cryptographic operations:

1. **Risk Assessment:** circomlibjs is used for circom circuit interactions
2. **Mitigation:** Ensure circomlibjs is only used for proof generation/verification, not key generation
3. **Alternative:** Monitor for circomlibjs updates that may switch to @noble/curves
