# ADR-004: Lean ↔ Rust FFI Bridge via lean-rs-host

## Status
Proposed

## Axis (Phase Mirror tension class)
urgency vs capacity

## Owner (multi-agent lever)
`the-examiner`

## Dissonance Score
- Impact = severity (3) x blast radius (12) = **36**
- Tractability = **2.0**
- **Score = 72.0**

## Context (stated intent vs implementation)
The documented intent below is not reflected by the current mathematical Lean 4
implementation. This is a measured gap produced by the Phase Mirror operational
loop.

### Stated intent (documents)
  - docs/README.md:18 — asserts [FFI bridge] “The Lean verified kernel
    communicates with the Rust runtime through a canonical, proof-preserving
    FFI bridge.”
  - docs/adr/ADR-003-sovereign-stack-implementation.md:24 — asserts [proof
    certificate portability] “Proof certificates generated in Lean must be
    serializable, transportable, and verifiable in the Rust runtime.”
  - docs/adr/ADR-004-UCC-blueprint-completion.md:61 — asserts [proof hash
    linkage] “Proof hashes from `RiemannHypothesis_from_F1_square` are embedded
    in `ContractivityReceipt.proof_hash` across all four surfaces.”

### Implementation reality (lean/)
  - FFI between `Prime/lean/` and `crates/` is currently ad-hoc: manual
    `lean::mk_array` / `lean::to_lean` calls in `crates/phase-mirror-edge/`
    without a canonical bridge specification
  - No serialization format for proof certificates between Lean and Rust
  - `ContractivityReceipt.proof_hash` is computed by hashing the raw `.olean`
    file bytes, which is fragile and non-reproducible across Lean versions
  - The `lean-rs-host` crate exists but is not integrated into the core bridge
    path; it is used only in experimental benchmarks in
    `crates/lean-rs-host-bench/`

### Manifested boundary
Leaked (unmanifested): YES — bridge architecture decision is NOT recorded in
any current ADR; the ad-hoc FFI is a silent leak risk for proof-certificate
integrity.

## Decision (the lever)
Resolve the dissonance by adopting `lean-rs-host` as the canonical FFI bridge
between the Lean 4 verified kernel and the Rust runtime. All communication
between `Prime/lean/` and `crates/` must flow through a single,
proof-certificate-preserving interface defined by the `lean-rs-host` protocol.
The bridge must serialize Lean proof terms into a deterministic, versioned
binary format (e.g., MessagePack with a fixed schema) that Rust can hash,
verify, and embed into `ContractivityReceipt` structures. No direct FFI calls
outside `lean-rs-host` are permitted in the sovereign-stack surfaces.

## Consequences
- **Positive**: proof certificates are transportable across language boundaries
  without losing kernel-checkability; the bridge is a single auditable surface;
  `ContractivityReceipt.proof_hash` becomes deterministic and reproducible
  because it hashes the serialized certificate, not the volatile `.olean` bytes.
- **Negative / Constraints**: `lean-rs-host` integration adds build complexity
  (native compilation, cross-language linking, MSRV alignment); the
  serialization format must be versioned to handle Lean version upgrades;
  runtime verification overhead adds latency to the contractivity path.
- **Verification Strategy**: `lake build` and `cargo build` must both succeed
  in a single CI job; the bridge must pass an end-to-end test that proves a
  theorem in Lean, serializes the certificate, deserializes it in Rust, hashes
  it, and verifies the hash matches `lake build` output; `scripts/test_ffi_bridge.sh`
  must pass on every commit to `Prime/lean/` or `crates/lean-rs-host/`.

## Metrics (resolution is confirmed when)
- All 4 sovereign surfaces (Chromium, VS Code, ESP32, local-first) use
  `lean-rs-host` for proof-certificate ingestion, with zero ad-hoc FFI calls.
- A proof certificate generated by `lake build` in `Prime/lean/F1/UCC.lean` can
  be serialized, hashed in Rust, and verified to match the Lean-side proof hash
  in under 500 ms on target hardware.
- `ContractivityReceipt.proof_hash` values are identical across two clean
  builds on different machines with the same Lean and Rust toolchain versions.
- `cargo kani` verifies that the FFI bridge does not introduce undefined
  behavior in the deserialization path.

## Actionable Levers
1. Integrate `lean-rs-host` into `Prime/lean/lakefile.lean` and
   `crates/phase-mirror-edge/Cargo.toml` as a workspace member with a pinned
   MSRV (Rust 1.75+, Lean 4.3+).
2. Define the proof-certificate serialization schema in `docs/lean-rs-host-schema.md`
   and implement MessagePack bindings in `crates/lean-rs-host/src/serialization/`.
3. Replace ad-hoc FFI calls in `crates/phase-mirror-edge/src/ffi/` with
   `lean-rs-host` client calls; add `scripts/test_ffi_bridge.sh` as a CI gate.
4. Update `docs/adr/ADR-003-sovereign-stack-implementation.md` to reference
   `lean-rs-host` as the mandatory bridge and remove ad-hoc FFI allowances.
5. Add a Kani model-checking suite in `crates/lean-rs-host/kani/` that verifies
   deserialization cannot panic or produce malformed proof hashes.
