# ADR-104: AI-Powered Formal Verification Agent (Ax-Prover / MerLean)

## Status
Proposed

## Owner (multi-agent lever)
`the-examiner`

## Dissonance Score
- Impact = severity (4) x blast radius (2) = **8**

- **Score = 5.6**  (Evolution plan rank 1 of 6)

## Context (stated intent vs implementation)
The current Lean4 verification is a static, human-driven process gated by
`lake build` in CI. The brief proposes extending it with an AI-powered formal
verification agent (Ax-Prover / MerLean) that analyzes unproven theorems and
proposes Lean4 proof sketches.

### Stated intent (documents)
- Brief "Deepen Formal Verification: From Proof to Autonomous Agent" — automate
  discovery and patching of vulnerabilities in the FPGA orchestrator and ZK
  circuits by proposing Lean4 proof sketches for unproven theorems.
- `docs/AGENTS.md` output contract requires every ADR to map to a verifiable
  Lean4 artifact; autonomous patching must not introduce unmanifested `sorry`.

### Implementation reality (lean/)
- Formal verification is human-gated; no autonomous proof-synthesis loop exists.
- `alp_sorry_manifest.json` is the only reconciliation point for known gaps and
  is not fed by any automated prover.
- No CI hook invokes a proof agent; proof debt is resolved manually.

### Manifested boundary
Leaked (unmanifested): NO — proposed as a net-new capability; begin life as
`Proposed` so the claim ("autonomous, continuous verification") is not asserted
until the agent's outputs are themselves verified artifacts.

## Decision (the lever)
Integrate a multi-agent LLM prover (Ax-Prover / MerLean class) into the CI/CD
pipeline as a **proposal-only, human-ratified** stage:

1. The agent scans `lean/` for unproven theorems and entries in
   `alp_sorry_manifest.json`, then emits Lean4 proof sketches.
2. Proposed proofs are validated by an isolated `lake build` in a sandbox
   branch. Only proofs that compile free of unmanifested `sorry` are eligible
   for a human review queue.
3. On human acceptance, the manifest is reconciled (entry removed) and a WORM
   ledger attestation records the patch + reviewer.
4. The agent never auto-merges; autonomy is bounded to *proposal synthesis*,
   preserving the sorry-bounded guarantee as a CI gate.

This transforms formal verification from a milestone into a continuous,
autonomous process while keeping every claimed guarantee auditable.

### Objective
Integrate an LLM-based theorem prover (Ax‑Prover or MerLean) into the CI
pipeline to propose Lean4 proofs for new code, reducing the human burden of
formal verification.

### Motivation
As the codebase grows, manual proof writing becomes the bottleneck. An AI agent
can generate candidate proofs for simple lemmas, freeing engineers to focus on
complex invariants (e.g. FPGA orchestrator and ZK circuit guarantees).

### Non-Goals
- Do not replace human review; all proposed proofs must be reviewed and approved
  before merge.
- Do not use AI to create new `sorry`s; the agent must only propose proofs that
  compile.

### Technical Approach
1. **Training/Fine‑tuning**: Fine‑tune a model on our existing `SQD.lean`,
   `Circuits.lean`, etc., as well as Mathlib4.
2. **CI integration**: On each PR, the agent:
   - Identifies unproven lemmas (marked `sorry` or `admit`).
   - Generates up to 3 candidate proofs.
   - Runs `lake build` on each; if a proof compiles, it is attached as a PR comment.
3. **Human‑in‑the‑loop**: An engineer reviews and accepts/rejects; accepted
   proofs are merged and the `sorry` is removed, reconciling
   `alp_sorry_manifest.json`.

### Blockchain Anchor (Invariant 5)
- **Problem**: If an AI proposes a Lean proof, how do we know the proof wasn't
  tampered with retroactively?
- **Implementation**: The CI computes the SHA‑256 hash of the newly generated
  Lean theorem (the `Expr` hash) and includes it in `ProofRegistry.sol`
  alongside the commit timestamp.
- **Result**: The mathematical process of patching the system is anchored. We
  can forever prove a specific theorem existed in the codebase on a specific
  date, making the audit trail of system upgrades legally binding. This root is
  folded into the daily aggregated anchor (ADR-110).

### Implementation Phases
- **Week 1–2**: Set up model inference environment (GPU runner) and draft prompt
  engineering.
- **Week 3‑4**: Integrate with GitHub Actions; proof candidates posted as PR
  comments.
- **Week 5‑6**: Implement feedback loop: correct proofs are used for
  fine‑tuning; performance metrics tracked.

## Consequences
- **Positive**: continuous vulnerability patching for FPGA orchestrator and ZK
  circuits; reduced manual proof debt; faster closure of Phase Mirror tensions.
- **Negative / Constraints**: LLM-proposed proofs require human ratification
  (no autonomous merge) to preserve soundness; adds CI compute surface and a
  review queue; proof sketches may be unsound and must be rejected by `lake
  build`, never silently accepted.
- **Verification Strategy**: each agent proposal must pass an isolated
  `lake build` with all `sorry` tracked in alp_sorry_manifest.json; ratified patches are recorded in
  the WORM ledger; the dissonance for the relevant subsystem trends to 0 on
  subsequent loop runs.

## Metrics (resolution is confirmed when)
- A CI stage exists that invokes the proof agent against `lean/` + the sorry
  manifest and emits candidate proofs to a review queue.
- At least one historical `sorry` entry is auto-proposed, builds cleanly, and
   is ratified with a WORM attestation.
- The sorry-bounded boundary is preserved: no agent output is merged without a
  compiling proof.
- Metric: percentage of new `sorry`s automatically filled within 24 hours
  (target > 60%).

## Actionable Levers
1. Add a CI job `scripts/run_proof_agent.sh` that runs the prover over
   `alp_sorry_manifest.json` and open PRs with Lean4 proof sketches.
2. Enforce an isolated `lake build` gate on agent PRs; block merge on any
   unmanifested `sorry`.
3. Extend the WORM ledger submitter (`scripts/submit_worm_attestation.py`) to
   record ratifications with reviewer identity + patch hash.

## Artifacts
- `scripts/ai_proof_agent.py` — CI proof-agent driver.
- `models/proof_model.pt` — fine-tuned prover weights.
- CI workflow update (GitHub Actions) posting proof candidates as PR comments.
- Metrics dashboard (`uac_ai_proof_attempts_total`, fill-rate panel).

## Dependencies
- Lean4 environment (`lake build` available in CI).
- GPU runner for inference / fine-tuning.
- Access to training corpus (`SQD.lean`, `Circuits.lean`, Mathlib4).

## Risk
- **Risk**: Model may produce unsound proofs that pass type‑checking but are
  logically invalid.
- **Impact**: High (could silently weaken a guarantee if merged).
- **Mitigation**: Human review is mandatory (no auto-merge); additional
  property-based tests on every accepted proof; `lake build` gate rejects any
  unmanifested `sorry`.
