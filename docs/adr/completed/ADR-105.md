# ADR-105: Batch ZK Proofs via STARK Aggregator

## Context (stated intent vs implementation)
The current Groth16 attestation is a per-run, single-proof model. The brief
proposes a STARK-based aggregator compressing thousands of FeMoco attestations
into one succinct proof (~100–200 KiB) to cut gas costs and on-chain load,
enabling a 10–100× throughput increase.

### Stated intent (documents)
- Brief "Extend ZK Attestation: Batch Proofs" — aggregate many attestations
  into a single STARK; preserve attestation completeness for every run.
- `circuits/` attestation circuits currently target single-run Groth16 proofs.

### Implementation reality (circuits/)
- Per-run Groth16 proofs are emitted per session; no batching/aggregation
  layer exists.
- No on-chain STARK verifier; gas cost scales linearly with run count.

### Manifested boundary
Leaked (unmanifested): NO — net-new capability; claimed throughput gain must
remain `Proposed` until the aggregation preserves the per-run attestation
completeness invariant (Constraint 3 of the evolution plan).

## Decision (the lever)
Implement a two-tier attestation pipeline:

1. **Per-run leaf proofs** (unchanged Groth16) continue to guarantee every
   production run is individually attestable — preserving Constraint 3.
2. **STARK aggregator** consumes N leaf proofs (or their commitments) and
   produces one succinct STARK (~100–200 KiB) over a Merkle/batch accumulation
   scheme, with an on-chain STARK verifier contract.
3. **Soundness + size bounds** are encoded as Lean4 theorems
   (`AggregatorSoundness`, `ProofSizeBound`) and discharged before the
   aggregator is trusted in production.
4. Batching is opt-in per batch window; single-proof fallback remains for
   low-throughput sessions so no run is left unattested.

This yields the 10–100× gas/on-chain-load reduction while keeping every run
cryptographically accountable.

### Objective
Aggregate multiple FeMoco attestations into a single STARK proof, reducing
Ethereum gas costs and on-chain storage by 10–100×.

### Motivation
Per-run Groth16 proof costs ~500k gas; at 100 concurrent runs, this is
economically unsustainable for high-volume clients. Batch proof reduces cost per
attestation to <5k gas.

### Non-Goals
- Do not compromise individual attestation verifiability; the batch proof must be
  decomposable into per-run verifications.
- Do not change the `AttestationRegistry.sol` interface; batch proof must be
  submitted as a single transaction.

### Technical Approach
1. **Accumulator construction**: Use a STARK (e.g., Winterfell) to prove that a
   list of Groth16 proofs, or directly the R1CS satisfiability for multiple
   runs, is valid.
2. **Circuit**: Compose a circuit that accepts `N` sets of public signals and
   verifies each individually via repeated pairing checks, but in a single
   polynomial proof.
3. **Sidecar upgrade**: The TypeScript sidecar accumulates proofs in a buffer
   (e.g., 10 runs or 5 minutes) and submits a batch transaction to a new
   `BatchAttestation.sol` contract.
4. **Smart contract**: `BatchAttestation.sol` verifies the STARK proof and emits
   aggregated events; clients can still query individual attestations via a
   merkle root.

### Blockchain Anchor (Invariant 5)
- **Problem**: Pushing 100+ attestations per minute to Ethereum makes gas
  costs prohibitive, forcing off-chain storage that breaks the "forever"
  promise.
- **Implementation**: The STARK aggregator submits a Merkle root of an entire
  day's worth of runs (thousands of FeMoco simulations) to
  `BatchAttestation.sol`. Only the root is verified on-chain; the full list of
  individual Q‑SQD hashes stays in IPFS/Arweave, cryptographically bonded to
  the root via recursive verification.
- **Result**: The blockchain eternally stores the cryptographic anchor. Anyone,
  at any future time, can download the off-chain batch and verify every run
  against the immutable on-chain root — proving the record is forever
  immutable at scale. This is the gas subsidy that funds Invariant 5 anchoring
  across all layers (ADR-110).

### Implementation Phases
- **Week 1‑3**: Prototype STARK circuit for 2–3 attestations; benchmark gas.
- **Week 4‑6**: Integrate into sidecar; add accumulation logic and buffering.
- **Week 7‑9**: Deploy `BatchAttestation.sol` on testnet; run end‑to‑end with
  mock data.

## Consequences
- **Positive**: drastic gas/on-chain-load reduction; throughput scales without
  proportional cost; per-run attestation completeness preserved.
- **Negative / Constraints**: larger proving time for aggregation; new
  on-chain verifier to audit; STARK transparency (no trusted setup) trades
  proof size vs Groth16 succinctness — bounded to the ~100–200 KiB target.
- **Verification Strategy**: `AggregatorSoundness` and `ProofSizeBound` prove
  the batch preserves per-run attestability and stays within size bounds; a
  `lake build` regression plus a Rust verifier test confirm the invariant.
