# ADR-106: Post-Quantum Provider Signatures (CRYSTALS-Dilithium)

## Status
Proposed

## Axis (Phase Mirror tension class)
risk claimed vs risk owned

## Owner (multi-agent lever)
`the-guardian`

## Dissonance Score
- Impact = severity (4) x blast radius (3) = **12**

- **Score = 9.6**  (Evolution plan rank 2 of 6)

## Context (stated intent vs implementation)
The sidecar's provider signature currently uses ECDSA. The brief proposes
transitioning to a NIST-approved lattice-based scheme (CRYSTALS-Dilithium) to
future-proof attestation against cryptographically relevant quantum computers
(CRQCs).

### Stated intent (documents)
- Brief "Post-Quantum Readiness" — replace ECDSA provider signatures with
  CRYSTALS-Dilithium for quantum-safe attestation.
- `crates/multiplicity-crypto/` holds the current signature primitives.

### Implementation reality (crates/multiplicity-crypto/)
- Provider signatures are ECDSA-based; no PQC primitive is integrated.
- Verification path assumes classical curve signatures end-to-end.

### Manifested boundary
Leaked (unmanifested): NO — net-new capability. Because this is a
**prerequisite for trusting autonomous actions on-chain**, it is scheduled
first in Phase A of the evolution plan. Claimed "quantum-safe" status remains
`Proposed` until Dilithium verify is proven equivalent in the attestation
pipeline.

## Decision (the lever)
Adopt CRYSTALS-Dilithium (NIST FIPS 203/204 family) as the provider signature
scheme for the sidecar:

1. Add a Dilithium sign/verify module to `crates/multiplicity-crypto/` with a
   Rust implementation (e.g. `dilithium-crate` or `pqcrypto-dilithium`),
   behind a feature flag to allow dual-signature coexistence during migration.
2. Encode the signature-verification soundness and the
   classical→PQC attestation-equivalence as Lean4 theorems
   (`DilithiumVerifySound`, `AttestEquivalence`) and discharge them before
   enabling by default.
3. Dual-sign (ECDSA + Dilithium) during a transition window; retire ECDSA once
   the governance test confirms Dilithium-only verification passes and the
   WORM ledger records the cutover.
4. Keep the on-chain verifier upgrade path explicit so the STARK aggregator
   (ADR-105) consumes Dilithium-signed leaves.

This future-proofs attestation against CRQC attacks while preserving
verifiability during migration.

### Objective
Add CRYSTALS‑Dilithium signatures alongside ECDSA for provider attestations,
future‑proofing against cryptographically relevant quantum computers (CRQCs).

### Motivation
ECDSA is vulnerable to Shor's algorithm. Dilithium is NIST‑approved and offers
security against quantum attacks.

### Non-Goals
- Do not remove ECDSA; maintain dual‑signature support for compatibility.
- Do not require clients to upgrade immediately; the contract can accept either
  signature.

### Technical Approach
1. **Sidecar integration**: Add Dilithium key generation (using `pqcrypto` Rust
   crate) and signing alongside ECDSA.
2. **Contract upgrade**: Extend `AttestationRegistry.sol` to accept a second
   signature field and verify against a registry of Dilithium public keys.
3. **Key management**: Store Dilithium keys in cloud KMS alongside ECDSA keys;
   rotate periodically.

### Blockchain Anchor (Invariant 5)
- **Problem**: ECDSA signatures securing today's transactions are vulnerable to
  Shor's algorithm in ~10–15 years. A "forever immutable" record that becomes
  unverifiable due to quantum decryption is effectively mutable.
- **Implementation**: The sidecar pushes dual signatures (ECDSA +
  CRYSTALS‑Dilithium) as part of the `BatchAttestation.sol` payload. The
  on-chain verifier checks both.
- **Result**: If a quantum computer emerges in 2040, the Dilithium signature
  remains cryptographically sound, ensuring 2026 FeMoco results stay legally
  and scientifically verifiable for the next century.

### Implementation Phases
- **Week 1‑2**: Integrate `pqcrypto` crate into sidecar; generate keys.
- **Week 3‑4**: Upgrade Solidity contract; add verification logic for Dilithium
  (using the `DilithiumVerifier.sol` generated by the reference implementation).
- **Week 5**: Enable dual‑signature attestations on testnet; monitor performance
  (Dilithium verification is ~10× slower than ECDSA, but acceptable).

## Consequences
- **Positive**: quantum-safe provider attestation; meets NIST post-quantum
  standardization; prerequisite for trusting autonomous on-chain actions.
- **Negative / Constraints**: larger signatures/inclusion cost than ECDSA;
  migration window requires dual-signature support; on-chain verifier must be
  upgraded in lockstep with ADR-105.
- **Verification Strategy**: `DilithiumVerifySound` and `AttestEquivalence`
  prove correctness and equivalence; a Rust+Lean regression test confirms
  Dilithium-only verification before ECDSA retirement.

## Metrics (resolution is confirmed when)
- `crates/multiplicity-crypto/` ships Dilithium sign/verify with a proven
  `DilithiumVerifySound` theorem in `lean/`.
- A governance test verifies Dilithium-signed attestations end-to-end and
  rejects forged signatures.
- ECDSA is retired only after the WORM ledger records the cutover and the
  dual-signature window passes.
- Performance: latency increase < 50 ms under dual-signature mode.

## Actionable Levers
1. Integrate a Dilithium crate in `crates/multiplicity-crypto/` behind a
   feature flag; add dual-sign path in the sidecar.
2. Prove `DilithiumVerifySound` / `AttestEquivalence`; manifest any gap in
   `alp_sorry_manifest.json`.
3. Add a migration test and a WORM attestation for the ECDSA→Dilithium cutover.

## Risk
- **Risk**: Dilithium verification gas cost may be higher than ECDSA.
- **Impact**: Medium.
- **Mitigation**: Offer both ECDSA and Dilithium; let clients choose (low-cost
  ECDSA vs high-security Dilithium).
