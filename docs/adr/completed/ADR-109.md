# ADR-109: System-Level Quantum-Classical Resource Management

## Status
Proposed

## Axis (Phase Mirror tension class)

`the-commander`

- **Score = 4.0**  (Evolution plan rank 6 of 6)

## Context (stated intent vs implementation)
The FPGA orchestrator currently manages qudit dimension shifting between
sessions. The brief proposes elevating it to a system-level
quantum-classical resource manager: a robust low-level quantum-operations
scheduler and generic resource manager that dynamically allocates resources not
just between sessions but between computation stages (error correction, state
preparation, measurement).

### Stated intent (documents)
- Brief "Deepen FPGA Orchestration: System-Level Resource Management" —
  elevate the orchestrator to a hybrid resource manager.
- `crates/ace-zk/`, `crates/sigma/`, and orchestrator crates manage `d`-shifting
  today.

### Implementation reality (crates/)
- Orchestrator performs per-session `d`-shifting only; no stage-level
  scheduler exists.
- No generic resource manager spans error correction / state prep /
  measurement stages.

### Manifested boundary
Leaked (unmanifested): NO — net-new capability. The orchestrator's actions
(real-time `d`-shifts from ADR-107, scheduling from this ADR) must remain
reconstructible (Constraint 4) and within the 100-qudit boundary (Constraint
2). "Self-optimizing" is `Proposed` until the scheduler is proven deadlock-free
and boundary-safe.

## Decision (the lever)
Promote the orchestrator to a system-level quantum-classical resource manager:

1. **Low-level QPU ops scheduler**: a generic scheduler allocates QPU time
   across stages — error correction, state preparation, measurement — not just
   across sessions.
2. **Generic resource manager**: a unified manager tracks classical FPGA and
   QPU resources, exposing a reservation API consumed by ADR-107's
   preemptive `d=8` shifts and ADR-108's active-space runs.
3. **Formal guarantees (inviolable)**:
   - `SchedulerDeadlockFree`: the stage-level scheduler never deadlocks.
   - `StageShiftWithinBoundary`: any resource reallocation keeps every active
     computation within the 100-qudit CAS(20,20) envelope (Constraint 2).
4. **Auditability**: every schedule decision + `d`-shift is recorded with
   inputs in the WORM ledger (Constraint 4) so the hybrid execution is
   reconstructible.

This yields a more efficient, flexible hybrid environment optimizing
classical-FPGA ↔ QPU interaction.

### Objective
Elevate the FPGA orchestrator to a full system‑level resource manager that
dynamically allocates resources across computation stages (error correction,
state preparation, measurement) and between sessions.

### Motivation
Current orchestrator only handles per‑session dimension shifting. Future
workloads may require resource sharing between stages, e.g., using idle qubits
for error correction while running a VQE.

### Non-Goals
- Do not redesign the FPGA hardware; work within existing capabilities.
- Do not impact the zero‑`sorry` formal guarantees; the resource manager must
  be verifiable.

### Technical Approach
1. **Resource abstraction**: Model the FPGA as a set of resources (qudits,
   control units, memory).
2. **Scheduler**: Implement a real‑time scheduler (in Rust) that partitions
   resources among concurrent tasks, respecting dependencies and priorities.
3. **Integration**: Replace the current simple HashMap with a resource‑aware
   allocator that can reserve resources for future stages.

### Blockchain Anchor (Invariant 5)
- **Problem**: Dynamic dimension shifting (d=16 to d=8) is currently ephemeral.
  If a fault occurs, it's hard to prove when and why the orchestrator made a
  resource decision.
- **Implementation**: The orchestrator publishes a state digest (hash of current
  session dimensions and thermal allocations) to a lightweight
  `OrchestratorState.sol` log every 10 blocks (or via a lazy aggregator).
- **Result**: Forensic analysis of hardware contention becomes permanently
  auditable on-chain, resolving disputes about SLAs and "who was using what
  qubit when."

### Implementation Phases
- **Week 1‑4**: Design resource model and scheduler; prototype in Rust.
- **Week 5‑8**: Integrate into `fpga_pulse.rs`; replace existing
  dimension‑shifting logic.
- **Week 9‑12**: Add formal proof (in Lean4) that the scheduler never deadlocks
  and respects resource bounds.

## Consequences
- **Positive**: finer-grained resource use; flexible stage-level scheduling;
  backbone for ADR-107 proactive shifts.
- **Negative / Constraints**: more complex concurrency surface; scheduling
  policies need tuning; deadlock/starvation must be ruled out by proof, not
  test alone.
- **Verification Strategy**: `SchedulerDeadlockFree` and
  `StageShiftWithinBoundary` prove safety; a simulation test exercises
  concurrent stage allocations and confirms no boundary breach.
