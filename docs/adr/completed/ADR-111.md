## Context (stated intent vs implementation)
The discovery of an *unconditional* path for breaking RSA is a critical event
that triggers the project's high-assurance defensive protocols. Because it
interacts with global cryptographic standards, it must be handled under the P²C
Core v1.1 governance mandate — specifically defensive publication and prior-art
disclosure — so the technique is irrevocably placed in the public domain and
cannot be patented or weaponized for private control.

### Stated intent (documents)
- `docs/SOVEREIGN_SYNTHESIS_DEFENSIVE_PUBLICATION.md` — frames the project's
  publications as defensive prior art; establishes the Sedona Spine Mandate and
  Fail-Closed (`SIG_GOV_KILL`) mechanics.
- `docs/SECURITY_ADVISORIES.md` — existing process for tracking cryptographic
  vulnerabilities and mitigation (elliptic/ECDSA precedent).
- `crates/multiplicity-crypto/docs/Multiplicity_Crypto*` — defensive
  publication purpose: prevent third-party patent claims.
- ADR-106 (CRYSTALS-Dilithium) — the post-quantum primitive the ecosystem
  transitions toward; ADR-110 (State Anchor) — the immutable priority-date
  record.

### Implementation reality (repo)
- Lean4 proofs are gated by the Sedona Spine sorry-bounded mandate; any break
  proof must be "audit-tight" before external disclosure.
- `contracts/BreakGlassEscrow.sol` already provides a verifiable break-glass
  disclosure/escrow primitive; `circuits/breakglass.circom` exists.
- No ADR yet codifies a Protocol-Level RSA break as a Constitutional L0 invariant
  update or activates an ACE fail-closed gate for RSA-based modules.

## Decision (the lever)
Treat the RSA break exactly like a Constitutional L0 invariant update, not a
press release. Codify it as a Protocol-Level Change with four gated phases:

1. **Internal Audit & Formal Certification (Glass Box)**
   - Formalize the break in Lean4 with sorry-bounded verification (Sedona Spine mandate);
     the proof must be computationally verified before any external body is
     informed.
   - Hash the proof artifact and commit it to the AuditChain / State Anchor
     (ADR-110 `ProofRegistry.sol`) to establish an immutable, timestamped
     **priority date**.
2. **Defensive Publication Strategy**
   - Pre-emptive prior-art filing with USPTO and international patent offices to
     void future patentability of the specific break.
   - Publish the full technical report as a **Defensive Publication** under the
     Prime Materia License (CC BY-SA 4.0) in the public repositories.
3. **Stakeholder Notification & Disclosure**
   - Submit to NIST Cryptographic Standards / IETF, framed as a post-quantum
     transition necessity (per the project's NIST AI RMF alignment).
   - Notify the Multiplicity research core (Citizen Gardens channels) to
     initiate the "Epochal Progression" review.
   - Submit a formal preprint to arXiv (cs.CR / math.NT).
4. **Governance Protocol**
   - This ADR codifies the break as a Protocol-Level Change requiring immediate
     review of all legacy RSA implementations in the Phase Mirror stack.
   - **Activate the Fail-Closed gate**: the ACE (Arithmetic Control Engine)
     runtime enters a `governance-review` state for every module relying on
     RSA-based signatures until transition to Dilithium (ADR-106) / post-quantum
     primitives is complete. On synthesis failure, `SIG_GOV_KILL` is enforced
     per the Sedona Spine Fail-Closed mechanics.
   - Leverage `contracts/BreakGlassEscrow.sol` + `circuits/breakglass.circom`
     for verifiable, escrowed disclosure of the finding.

## Consequences
- **Positive**: the technique is irrevocably public-domain (unpatentable,
  non-weaponizable); ecosystem gets a coordinated, non-vacuum transition path to
  prime-indexed / post-quantum primitives; priority date is immutable.
- **Negative / Constraints**: disclosure is legally sensitive and partially
  outside repo control (patent offices, arXiv, NIST timelines); ACE fail-closed
  may degrade RSA-dependent modules until migration completes; requires the
  Lean4 proof to be audit-tight first.
- **Verification Strategy**: `RSABreakDefensiveDisclosure` and
  `FailClosedOnRSA` prove (a) the disclosure record is anchored and public and
  (b) every RSA-dependent module is gated to `governance-review` until
  transition; a test simulates the gate and confirms `SIG_GOV_KILL` on
  synthesis failure.

## Implementation Checklist (Fail-Closed Gate)

1. **Lean formalization** — `lean/SNAPKITTY/SnapKitty/SecurityGate.lean` defines
   `SecurityMode`, `SecurityState`, `is_lawful_operation`,
   `validate_contraction_gate`, and the provable
   `rsa_governance_fail_closed` invariant (Review ∧ RSA ⇒ ¬ lawful). The
   snippet's original `rsa_governance_fail_closed` is **not provable as
   written** — `decide` on a `match` returning `False`/non-`False` does not
   `simp`, and `False.elim` has no target — so the theorem is stated directly on
   the predicate.
2. **Rust guardian** — `crates/ace/src/lib.rs` adds `SecurityState` /
   `SecurityMode` and `validate_contraction_gate` (the formal mirror of the
   Lean gate). The ACE daemon enters `GovernanceReview` on the ADR-111 trigger
   and verifies the gate before executing any operator word; on violation it
   raises the Fail-Closed response (`GateError::SecurityViolation`, i.e.
   `SIG_GOV_KILL`) per `SOVEREIGN_SYNTHESIS_DEFENSIVE_PUBLICATION.md`.
3. **WORM audit log** — every `Err(GateError::SecurityViolation)` is logged to
   the `crates/security` `AuditLogger` (trace_id + timestamp + action +
   result + severity) so the breach is reconstructible (Constraint 4 / ADR-110).
4. **State sync** — when the daemon enters `GovernanceReview`, `SecurityState`
   in the runtime is updated immediately and the gate is re-checked on every
   subsequent operator word.
5. **Authoritative checker** — `validate_contraction_gate` is the single source
   of truth, exposed for the FFI/binding layer; the ACE invariant set calls it
   so both the Lean proof and the Rust runtime enforce the same predicate.

### AttestationHash / state-spoofing hardening (open)
The gate trusts the in-memory `SecurityState`. To prevent spoofing of the mode
or `primitive_id`, bind `SecurityState` into the CRMF telemetry path
(`crates/engine/src/crmf_binding.rs`, `LawfulRecursionHash`) and require the
gate to verify an `AttestationHash` over the state before acting. This is
recommended but not yet implemented; track under `alp_sorry_manifest.json` if
shipped as a separate proof obligation.
