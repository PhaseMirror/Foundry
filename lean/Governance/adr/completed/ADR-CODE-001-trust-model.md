# ADR-CODE-001: Local Coding Agent Trust Model

## Status
Proposed

## Context
The Commander ecosystem is evolving from an operator-led shell to a governed coding-agent architecture. To maintain privacy, avoid SaaS costs/quotas, and ensure data residency, a local LLM stack is required. However, introducing an LLM agent creates a new trust boundary. We must define how this agent interacts with the repository and what authority it holds compared to the Commander core.

## Decision
We adopt a "Planner-Governor" separation of concerns. The coding agent (e.g., OpenCode, custom MCP server) is strictly a **Planner**, while the Commander is the **Governor and Executor**.

1. **Planner Role**: The coding agent may read repository context (as provided by Commander) and propose plans, patches, and explanations. It has **no direct authority** to mutate the filesystem, execute shell commands, or interact with Git.
2. **Governor Role**: The Commander (via `pscmd`) is the sole entity authorized to execute mutations. It validates all agent proposals against the Atomic Language Policy (ALP), executes them via the Sigma kernel, and records every action as a `UnifiedWitness`.
3. **Registry Requirement**: All components of the coding stack (Inference server, Planner agent) must be registered in `state/mcp_registry.yaml` and attested via binary hash. Commander will refuse to interact with any component that fails attestation.
4. **Inference Boundary**: The local LLM server (e.g., Ollama) is treated as an internal utility. It is not a policy boundary; governance logic must remain in the Rust-based Commander core.

## Consequences
- **Privacy & Control**: No source code or telemetry leaves the local machine.
- **Auditability**: Every AI-suggested change is logged, witnessed, and attributable.
- **Flexibility**: The underlying LLM or Planner can be swapped (e.g., Ollama for LocalAI, or OpenCode for a custom Rust agent) without changing the core trust model.
- **Safety**: "Hallucinated" or malicious commands from the LLM are caught by the ALP gate before execution.

## Verification
- `pscmd mcp list` must show the coding agent and inference server with `governed: false` (since they are planners) but `trust: internal`.
- Integration tests must confirm that patches generated by the agent are only applied after Commander validation.
- The `UnifiedWitness` for a coding session must include the model identity and the specific plan/patch that was approved.
