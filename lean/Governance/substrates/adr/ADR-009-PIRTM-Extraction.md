# ADR-009: PIRTM Extraction Pipeline (Lean-to-Rust)

## Status
Proposed

## Context
The MOC v2 architecture mandates a "Lean-First" specification, where PIRTM transitions, stability proofs, and resonance gates are formal theorems in Lean 4. However, runtime execution (numerical kernels, tensor ops) must occur in Rust for performance.

## Decision
We implement a formal extraction bridge that binds every Rust executable artifact to a Lean-proved stability witness.

### Mechanism

1.  **Lean Artifact Binding:** Each `Transition` morphism in Lean includes a `ProofHash`, generated by the Lean 4 proof-term hash at compilation time.
2.  **Extraction Bridge:**
    - The Lean compiler (`pirtm-compiler`) generates the Rust representation of the morphism.
    - The generator embeds the `ProofHash` as a constant in the generated Rust code.
3.  **Runtime Integrity:**
    - At runtime, the Rust executable calculates the hash of its own transition logic.
    - It verifies this against the embedded `ProofHash` and the verified manifest provided by the verifier enclave (ADR-008).
4.  **CI Enforcement:**
    - CI/CD build scripts perform a final hash-link verification before the Rust binary is tagged for production staging.

## Verification
- Lean specification is the sole source of truth for PIRTM stability.
- Rust execution is cryptographically and formally linked to the Lean stability certificate (ACE compliance + Resonance Bound).
- Any runtime transition not traceable to a Lean-proved `ProofHash` triggers an immediate shutdown.
